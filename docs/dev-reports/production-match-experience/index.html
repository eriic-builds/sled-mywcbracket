Safety boundaries kept intact

Your bracket stays local

Picks, rivals, favorites, themes, and manual score views remain in browser storage. The site has no account database.

Old share links still work

The existing #b= base64url wire format was not replaced or version-bumped.

Preview state is isolated

Demo and shared brackets stash owner what-if scores once, then restore them when the owner returns.

Portrait requests follow intent

The initial page makes no portrait-host request. Hover intent or an explicit open action starts the credited external portrait.

Fallback results stay valid

When FIFA is unavailable, football-data.org results create honest partial detail records instead of breaking the workflow commit.

Every render has an off switch

Document handlers, window handlers, timers, frames, observers, connectors, dialogs, and trophy scenes are cleaned before rerender.

How the work was executed

Phase 1 - Analyze both codebases

The sandbox defined accepted behavior. SLED defined production truth: UTC result timestamps, five-field kickoff tuples, dense sync windows, local rivals, share links, and preview isolation. The port used production contracts rather than copying the sandbox repository wholesale.

Phase 2 - Build a tested tournament model

bracket-tree.js derives 31 nodes, 30 feeder edges, branch sides, columns, and slots from ko_feed. Both visual layouts read the same model.

Phase 3 - Port the visual experience

The renderer gained complete match cards, Actual replacements, placeholders, final center stage, Sideways parity, mini overview, trophy mounts, readable future dates, and accessible match targeting.

Phase 4 - Add FIFA facts and portraits

The sync writes deterministic details. The browser adds quick facts, an accessible dialog, credited portraits, one-frame movement between surfaces, and responsive direct-link fallback.

Phase 5 - Protect production social behavior

Dashboard rerenders now teardown every feature lifecycle. Shared and demo views preserve the owner's local state. Owner-only controls remain hidden outside owner mode.

Phase 6 - Review, measure, and release

Hermetic JavaScript and Python suites, generated-data validators, a live FIFA dry run, visual parity measurements, privacy checks, responsive checks, and browser lifecycle checks form the release gate.

Learning moments

1. Port behavior, not files

The sandbox and production app look related, but production owns newer social and data rules. Copying whole files would erase those rules. A good port moves the accepted experience through the destination architecture.

ELI5: Move the recipe into the restaurant's kitchen. Do not replace the kitchen.

2. One truth should own identity

Scores and decider notes already have a canonical result record. Detail feeds enrich that identity. They must not silently redefine it.

ELI5: The scoreboard decides who won. The newspaper adds the story.

3. Every setup needs teardown

A rerendered app must remove global handlers, observers, timers, animation frames, and GPU work. Detached HTML is not enough when window and document still hold callbacks.

ELI5: Turning off a room light does not unplug the old extension cord.

4. Privacy is about timing too

An external portrait can be useful and credited while still waiting for user intent before making a request. Initial load remains local. The external request begins when the user asks for the portrait.

ELI5: Do not call the museum until the visitor asks to see the exhibit.

5. Fallbacks need honest states

A backup result source lacks FIFA's full event detail. A labelled partial record is safer than fake completeness or a workflow failure.

ELI5: Write "summary" on a short report. Do not pretend it is the full book.

6. Visual parity needs measurements

A screenshot alone cannot prove no overflow, no leaked state, correct storage restoration, or zero hidden requests. Browser assertions turn those claims into evidence.

ELI5: Looking straight is not the same as checking with a ruler.

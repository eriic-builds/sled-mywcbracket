2. Project Contract and Audit Rules

This was not a generic "make it faster" pass. It was a bounded investigation with product, architecture, lifecycle, and evidence constraints.

Runtime contract

  • Static GitHub Pages site.
  • Vanilla HTML, CSS, and ES modules.
  • Locally vendored Three.js r170.
  • No backend, build step, worker, CDN, or new dependency.
  • Existing local storage and privacy behavior unchanged.

Behavior to preserve

  • Four-click hidden activation and exact bracket curtain.
  • Thirty-second deterministic ceremony and continuous final flares.
  • Seven diverse premium clay players and two-hand trophy contact.
  • Reduced motion, fallback, context loss, replay, Skip, and restoration.
  • Zero celebration scene work before valid activation.

4. Codebase Discovery

Dormant boundary

champion-celebration-trigger.js owns only the click counter and lazy boundary. Three clicks remain a true near miss.

Active controller

champion-celebration.js owns stage, reusable canvas, deterministic time, pause/resume, fallback, teardown, and restoration.

Heavy scene

champion-celebration-scene.js, models, and effects own seven rigs, stadium, crowd, trophy, IK, particles, shadows, and render calls.

What the trace changed about the investigation

  1. Layout and Paint were not the problem. The dominant work lived inside the celebration RAF callback.
  2. Renderer bypass collapsed p95 to 16ms in the baseline and about 13ms after batching. That ruled out broad controller rewrites as the main answer.
  3. Scene submission was fragmented. Baseline inventory showed 667 meshes, about 626 calls, and 659 shadow participants.
  4. Most player detail was rigid below existing animated parents. That created a safe batching boundary without crossing a joint or rig.
  5. Different colors blocked ordinary material-key batching. Player-local vertex colors provided a bounded consolidation path.
  6. Cold and steady cost moved differently. Direct buffer construction protected steady state but increased one-time work, which is reported separately.
Important discovery: the final scene still submits about 893k triangles. The optimization reduces CPU-side object, material, matrix, shadow, and draw fragmentation; it does not disguise a visual downgrade as performance work.

5. Architecture Overview

The architecture keeps the primary product cheap, then pays for the hidden reward only after the valid trigger. The optimization remains inside the scene/model ownership boundary.

Safe ownership boundary

A mesh may merge only with direct leaf siblings under the same animated parent. Parent motion still moves the final merged geometry exactly as before.

Unsafe boundaries retained

Separate joints, players, skinned or instanced meshes, morph targets, multi-material meshes, custom depth materials, and animated thumbs stay independent.

6. Dependency and Data Flow Analysis

Activation and lifecycle

Per-frame trophy and arm order

  1. Apply deterministic rig pose and root travel for current logical time.
  2. Place and rotate the scene-space trophy carrier from captain world position and quaternion.
  3. Read named left and right grip anchors from the trophy.
  4. Solve each arm with scoped ancestor-chain matrix updates.
  5. Update crowd/effects and submit the scene once through the renderer.
Why order matters: reading grip anchors before trophy rotation mangles the hands during travel. Solving arms before carrier placement creates a one-frame lag. A broad full-scene matrix update fixes symptoms but repeats unnecessary work.

9. Risks and Edge Cases

Animation boundaries

  • Never merge across bones, player roots, or named articulated pivots.
  • Animated thumbs are explicitly protected; fingers stay below independent Groups.
  • Backward scrub must restore trophy rest quaternion and deterministic poses.
  • Skipped or reordered matrices can produce a one-frame grip lag.

Geometry and material correctness

  • Every vertex-colored replacement geometry needs a valid normalized color attribute.
  • Normals must use the normal matrix, not the position matrix.
  • Indices must be offset without overflowing the selected integer type.
  • Cross-player material sharing would erase deterministic appearance ownership.

Disposal and replay

  • Prune only geometry/materials no longer referenced by the scene.
  • Dispose replacements and retained originals exactly once.
  • Keep one reusable celebration canvas/context across replay.
  • Connected stage/canvas counts matter more than generic inspector-retained Attr nodes.

Evidence limits

  • CPU throttle is not physical-device thermals or GPU proof.
  • Headless Edge does not establish Safari/iOS behavior.
  • Transform use does not prove compositor promotion.
  • Cold, isolated, warmed, and full-matrix traces answer different questions.

11. Reasoning Trace Summary

This is a concise engineering decision log, not private chain-of-thought. It records the observable inputs and reviewable decisions another engineer needs.

Keep Plan 02 verification-only

Observation
The dashboard statically owns only a small trigger boundary and stylesheet.
Evidence
Dormant and three-click scenarios requested no heavy module and created no celebration resources or RAF.
Reasoning
A more complex loader would optimize an already-green boundary and add failure states.
Decision
Make no runtime loading change.
Tradeoff
Retain tiny first-load bytes for a simpler reliable activation.
Lesson
Zero-cost rhetoric should not replace measured materiality.

Remove broad matrix propagation

Observation
Trophy placement and each arm solve forced broad matrix updates.
Evidence
The scene already reads world transforms through methods that update required ancestor chains.
Reasoning
Repeated full traversal was redundant and did not improve grip correctness.
Decision
Rely on scoped world reads in the protected update order.
Tradeoff
Requires stronger order tests because broad updates can hide sequencing bugs.
Lesson
Correct dependency ordering beats brute-force synchronization.

Batch rigid player detail

Observation
Most detail meshes were rigid leaf siblings below animated parents.
Evidence
Baseline inventory showed 667 meshes and about 626 renderer calls.
Reasoning
Same-parent leaf merging preserves parent animation while reducing submission.
Decision
Use strict eligibility plus player-local vertex colors.
Tradeoff
Add one-time construction and material normalization.
Lesson
Batch below one animation owner, never across it.

Stop after bounded batching

Observation
Final focused renderer bypass is 12.865-13.315ms, while renderer-active p95 remains 69.262-75.374ms and a broader pass reached 118.413ms.
Evidence
DPR, shadows, static stadium batching, and render skipping lacked acceptable leverage.
Reasoning
More small JavaScript edits cannot close a renderer-dominated gap.
Decision
Document the limit and require explicit authorization for a rig/render architecture rewrite.
Tradeoff
Keep a red synthetic gate rather than weaken approved fidelity.
Lesson
Knowing when to stop is part of performance engineering.

12. Learning Mode: ELI5 and Senior Review

ELI5

Imagine the browser is a stage manager. Before the secret show starts, the manager carries no scenery at all. Once the show starts, the manager used to hand the renderer more than six hundred separate props every frame.

We tied safe groups of small decorations together, so the manager now hands over about three hundred props. The show still has the same players, stadium, trophy, confetti, and flares. It is much easier to manage, but the full set is still very large for the slow-phone test.

What a senior engineer notices

  • The fastest dormant feature is the one that has not allocated its world yet.
  • Draw calls, shadow passes, materials, and object traversal can dominate even with no Layout.
  • Scene graphs are ownership graphs; batching and disposal must be designed together.
  • A visual contract can be a stronger constraint than a synthetic budget.
  • A local proxy result should never be promoted into cross-browser certainty.
  • A failed target is still useful when attribution identifies the exact architecture boundary.
Core mental model: performance is not one number. It is a set of costs attached to lifecycle states: before activation, first construction, steady animation, hidden pause, replay, and teardown.

13. Recommended Decision, Execution Order, and Lessons
Recommended repository decision: treat the celebration and bounded performance changes as a commit candidate with one explicit known limit. Behavior, dashboard isolation, fallbacks, replay, restoration, and tests are ready. Do not label the active scene as meeting the 16ms 6x-phone budget.

If the 16ms target is advisory

  1. Review the opened dashboard, choreography surface, and this report.
  2. Commit the celebration plus bounded performance work together.
  3. Run Chrome, Firefox, Safari/iOS, and physical Android follow-up gates.
  4. Use real device evidence to decide whether a larger renderer rewrite is worth it.

If the 16ms target is a hard release gate

  1. Do not claim release readiness.
  2. Authorize a separate rig/render architecture brief.
  3. Prototype authored rigid skinning or lower-submission avatars behind visual comparisons.
  4. Re-baseline cold and active cost before replacing the current implementation.

Lessons learned

  1. Measure inactive and active features separately.
  2. Attribute before editing; renderer bypass was the pivotal experiment.
  3. Remove redundant synchronization before reducing visual content.
  4. Batch only inside a clear animation and disposal owner.
  5. Keep cold-start and steady-state evidence in separate tables.
  6. Report full-matrix outliers, isolated repeats, and warmed runs without substitution.
  7. Stop when the next step crosses from bounded optimization into product architecture.

How a browser draws movement

The browser measures, paints, then moves the finished layers.

The rendering pipeline is simplified here, but the distinction is enough to reason about this refactor.

1

Layout

Measure every box and decide where it belongs.

Geometry-affecting changes can trigger style recalculation and layout for the changed element and related descendants.

left top width height

2

Paint

Color the pixels again.

Paint records visual output. Animated shadows, filters, and some backgrounds can repaint even when layout is stable.

box-shadow filter

3

Composite

Slide or fade a finished picture.

Compositor-eligible transforms and opacity can reuse painted layers, subject to browser promotion and memory decisions.

transform opacity

The core comparison

More work

  • left: 2px to left: 18px
  • width: 20% to width: 70%
  • Writing touch-target width, height, left, and top every frame
  • Animating a growing box shadow forever

Preferred motion

  • transform: translateX(16px)
  • transform: scaleX(.7)
  • Resize once, then move with one transform
  • Pulse a pseudo-element with scale and opacity

Important myth check

A transform is a better request, not a magic GPU button. The browser still decides how to draw it.

Compositor eligible is not compositor guaranteed. JavaScript calculations remain on the main thread. Layer promotion depends on paint dependencies, device limits, browser heuristics, and memory pressure. Verify with the Performance and Layers tools.

Eight learning moments

Use these ideas when you inspect future motion changes.

1. Layout

Changing box size or coordinates can make the browser measure again.

Repeated width, left, top, and height writes are risky in event-frequency or frame-frequency paths.

2. Paint

A changing glow can recolor pixels every frame.

Animated box-shadow is paint-heavy. A prepared ring with transform and opacity is easier to reuse.

3. Composite

Slide or fade a finished picture when possible.

Transforms and opacity are compositor-eligible because they do not change document geometry.

4. RAF

A screen clock is not proof useful work happened.

A RAF callback may return early. Count physics, render, WebGL, and draw work before claiming a frame cap.

5. Sleep

A still scene should stop asking for another turn.

The ball pit requests no new module frame after 45 consecutive settled physics frames.

6. Frame pacing

Untouched decoration does not need every screen refresh.

Timestamp gates bound coarse ball work and passive trophy work while direct input bypasses the passive budget.

7. Reduced motion

Less motion is a user setting, not a broken animation.

Pseudo-elements need explicit reduced-motion coverage because stopping the parent animation does not target a new child effect.

8. Eligibility

A transform asks for the faster road. The browser picks the road.

Do not promise a separate GPU thread or force permanent layers. Promotion depends on each browser and device.

What changed in the code

Select a hotspot to compare the old request with the completed implementation.

Landing soccer-ball frame loop

The touch circle now slides without rebuilding its box. The ball engine sleeps after a real collision pile becomes still.

landing-ballpit.js caches dimensions in resize, writes one translate3d(), gates coarse work at 30fps, and stops requesting module frames after 45 settled physics frames.

Before style.left = x + "px"
style.top = y + "px"
After style.transform = "translate3d(...)"
sleeping = settledFrames >= 45

Why not CSS only?

CSS cannot model pair collisions, pointer impulses, bounded gravity, or flag decals attached to each ball's 3D matrix.

Team stat-card pointer tracking

Fast mouse events now share one screen update. Old events do not queue extra card moves.

interact.js measures on open or resize, stores only the newest pointer coordinates, and coalesces placement into one RAF callback with the individual CSS translate property.

Before card.style.left = x + "px"
card.style.top = y + "px"
After if (!cardFrame) nextFrame(placeCard)
card.style.translate = x + "px " + y + "px"

Composition detail

Individual translate preserves the existing entrance transform. The two effects compose instead of overwriting each other.

Score and builder progress bars

The bars stay full size and reveal progress by scaling from the left. Their boxes no longer resize during updates.

Score and builder fills use clamped zero-to-one scaleX() values with a left transform origin.

Before bar.style.width = percent + "%"
After bar.style.transform = "scaleX(" + ratio + ")"

Golden safety rail

The byte-locked fixture changed only from width:33% to the reviewed transform:scaleX(0.3375) representation.

Dashboard toggle knob

The knob stays anchored and slides visually to the same endpoint.

How the Technical Taste Council shaped the architecture

No one voice controls every choice. Each voice owns its decision domain.

Russinovich

Look under the hood and use a meter before tuning.

Measure Layout, Paint, long tasks, RAF, contexts, and compositing before optimization.

Karpathy

Use the smallest tool that solves the real problem.

No framework, worker rewrite, physics package, or animation abstraction.

Naval

Keep the fun people value. Remove waste around it.

The approved ball pit and trophy earn their complexity, so preserve and budget them.

Sean Grove

The recipe matters as much as the cake.

Six bounded specs prevent later models from re-deriving or expanding intent.

Willison / Hamel

Do not trust a fix until you try it.

Source tests plus real browser traces verify the implementation claim.

Yegge

Review one box at a time, not the whole warehouse.

Sequential plans and local handoffs keep each diff reviewable.

Geoffrey Litt

Future you should understand how to change it.

Keep logic in existing focused modules with explicit names and no specialist-only layer hacks.

Hanselman

Name every switch so the next person knows what it does.

Use searchable frame-budget, sleep, and lifecycle names. Update the false "VERBATIM" comment.

Context Engineering

Give each worker only the recipe and ingredients it needs.

GOAL
Keep approved motion and interaction quality while reducing unnecessary Layout, Paint,

PROJECT CONTRACT
- Runtime: [VANILLA / FRAMEWORK / OTHER]
- Supported browsers and devices: [TARGETS]
- Build and dependency constraints: [CONSTRAINTS]
- Features and visual behavior that must remain: [PRESERVE]
- Persistence, privacy, and reduced-motion rules: [RULES]

AUDIT BEFORE EDITING
1. Inventory every CSS animation and transition, JavaScript frame loop, pointer-following
   element, sliding menu, modal, progress fill, canvas, and WebGL scene.
2. Record a controlled baseline in each target browser. Use the same viewport, device pixel
   ratio, CPU throttle, interaction script, and recording duration before and after.
3. Measure Layout, Paint, scripting, long tasks, RAF activity, GPU or graphics work, canvas
   count, and WebGL context lifetime where available.
4. Find animation-time writes to left, right, top, bottom, width, and height. Separate real
   frame-loop hot paths from one-time static placement.
5. Find layout reads inside frame loops, unbounded requestAnimationFrame loops, hidden or
   settled scenes that keep rendering, broad transition rules, and paint-heavy animated
   shadows or filters.

PLAN BEFORE REFACTORING
Create a ranked plan set. Each plan must state the goal, exact files, implementation order,
edge cases, tests, browser evidence, and acceptance criteria. Measure first. Change one
bounded area at a time. Run overlapping plans sequentially.

IMPLEMENTATION DIRECTION
- Prefer transform and opacity for visual motion. Treat them as compositor-eligible, not a
  guarantee of GPU execution or a separate browser thread.
- Keep layout geometry stable during motion. Use translate or scale for pointer followers,
  sliding surfaces, toggle knobs, and progress fills where behavior allows.
- Coalesce high-frequency input into at most one DOM write per animation frame.
- Cache dimensions outside hot loops and refresh them on explicit resize or content-change
  events.
- Replace repaint-heavy effects with prepared layers animated through transform and opacity
  when the visual result stays equivalent.
- Give every loop pause, resume, sleep, wake, visibility, and teardown rules.
- Stop requesting frames when a simulation is settled. Wake only for real triggers.
- Budget passive canvas and WebGL rendering. Keep direct input responsive.
- Dispose WebGL resources and contexts when a responsive fallback replaces the scene.
- Preserve prefers-reduced-motion and any in-product motion control.
- Do not add blanket will-change rules. Permanent layers also consume memory.

VERIFICATION
1. Repeat the exact baseline scenarios after each bounded change.
2. Compare source guards, unit tests, snapshots, browser behavior, and traces.
3. Report raw before-and-after values. Do not claim a percentage when the scenarios do not
   match.
4. Separate cold-start cost from steady interaction cost.
5. State limits honestly. Headless traces do not prove every physical device, and transform
   use does not prove compositor promotion.

DELIVERABLES
- A brief that records the project contract and technical corrections.
- A measured BASELINE.md.
- Ranked, self-contained execution plans.
- Focused regression tests for the identified hot paths and lifecycle rules.
- A RESULTS.md with exact changes, matched measurements, validation, and remaining costs.
- A no-build interactive report when the repository supports static HTML documentation.

ACCEPTANCE GATE
- Approved motion and controls still behave correctly.
- No unapproved dependency, build step, storage key, or architecture change.
- No repeated layout writes in identified animation hot paths.
- Hidden or settled continuous scenes perform no useful frame work.
- Responsive fallbacks release resources they replace.
- Direct interactions stay below the agreed long-task and p95 main-thread budgets.
- All existing tests, snapshots, reduced-motion checks, and target-browser checks pass.

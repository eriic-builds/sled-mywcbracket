Chapter 1

The problem — we all had brackets, no way to compare

The World Cup started. The team had a shared bracket workbook — each person picked their winners, calculated points, tracked their champion. But it was all siloed. Everyone's dashboard lived on their own machine. There was no way to see how you stacked up against a coworker without emailing spreadsheets around.

What already existed

my-wc26-bracket — a Python tool that reads your Excel bracket, scores it against live results, and generates a personal dashboard. One person, one machine, no sharing.

It was great for your own view. But brackets are more fun when you can compare.

"Could I do like a share link that adds that user into your dashboard so we're not compromising everyone who doesn't want it to be surfaceable?"

That single question unlocked the whole design. The constraint was tight: it had to be free (no backend bills) and privacy-first (coworkers' real names should never appear anywhere they didn't choose).

Chapter 2

The insight — the URL is the database

💡

Your whole bracket fits in a link

31 games. 31 binary choices (win/lose). That's 31 bits of information — plus your display name and a tiebreaker score. It all fits in about 140 characters. You can encode it directly into a URL and send it like a text message.

Every share link looks like this:

https://eriic-builds.github.io/sled-mywcbracket/#b=eyJ2IjoxLCJiIjoiMDEwMTEwMDEw...

The part after # is your entire bracket — encoded, self-contained, requiring no server to decode. The recipient's browser unpacks it directly.

🔒 Structural privacy

The #fragment part of a URL is never sent to any server — that's how HTTP works. Not even GitHub's logs see it. Your picks and name literally cannot leak from the infrastructure.

💸 Zero cost

No database. No API. No auth. No serverless functions. Just a static HTML file on GitHub Pages — which is free forever. The link is the storage.

The honest bumps 🪨

Things that went wrong and what they taught:

  • GitHub Pages wouldn't deploy — the workflow token can configure Pages but can't create it. Fixed by enabling it via the API once, manually.
  • The sync bot pushed mid-session — git rejected the push. Fixed with git pull --rebase.
  • The golden test would break every day — it was reading live results that change 3× daily. Fixed by freezing a copy of the inputs.
  • Demo viewing deleted what-if edits — a silent data loss bug. Fixed with a stash/restore pattern.
  • Blank cells in the bracket map — 8 blank cells when all picks bust. The old code derived "who advanced" from the pick tree (dead-ends when picks bust). Fixed by reading directly from real results.
Chapter 4

Demo guide — show this live in 5 minutes

Open eriic-builds.github.io/sled-mywcbracket and walk through these steps.

Drop your bracket Excel — or tap "Try the demo"

The whole bracket parses and renders in your browser. Nothing leaves your machine. Notice the live scores — real results, synced automatically from FIFA's feed 3× a day.

Click "🔗 Share link" — edit the "Share as" name

Type your initials in the alias field. Watch the URL in the box update instantly — your bracket is encoding into that URL in real time. Copy it.

Open the link in a new window (or have someone else open it)

The full live-scored dashboard renders immediately — no login, no loading spinner, no account. Click "➕ Add to my leaderboard." Their bracket is now on your board.

Click "🏆 Leaderboard"

See both brackets ranked by confirmed points. Click a rival row to see the "where we differ" diff — undecided games only, sorted by how many points are at stake.

Switch to "Actual path" on the bracket map

Toggle between "My picks" and "Actual path." The actual view reads every slot from real results — blue triangles (▲) show where the real winner differs from your pick. Undecided slots show "Winner M97"-style placeholders.

🔐

Privacy talking point

Ask: "Where does your data go when you open a share link?" Answer: nowhere. The #b=... fragment is decoded entirely in the browser. GitHub's servers never see it. No cookie, no log entry, no database row. The only thing that touches a server is fetching the static HTML file and the JSON results — same as loading any webpage.

The Technical Taste Council

Every non-trivial decision was filtered through a set of named engineering voices — Karpathy (simplest thing that works), Hanselman (name things for clarity, no silent failures), Russinovich (know what's underneath), Naval (does this earn its complexity?), and others. It gave the AI a consistent judgment framework across sessions instead of making arbitrary calls.

Example: three attempts at positioning the 🏠 Home button. First try was wrong (margin-right: auto pushed everything right). Second try added a wrapper div (unnecessary complexity). Third: Karpathy + Litt say just put Home first in the flat centered group — that's enough. One line change.

🎯

The real lesson

AI-assisted development doesn't mean "AI writes everything." It means you stay at the level of outcomes — what should this do, what are the constraints, does this feel right — while the AI handles the implementation details, finds the edge cases, writes the tests, and explains the tradeoffs. The human's job shifts from typing code to making decisions.

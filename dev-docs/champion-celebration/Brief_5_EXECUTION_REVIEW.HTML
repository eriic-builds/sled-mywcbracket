Executive Summary

The decisive findings, recommendation, and first move.

Recommendation

Execute seven plans in order and begin with Plan 01. Do not start with character modeling. The highest-leverage risk is not art quality; it is opening a body-level modal from one of four simultaneously rendered bracket variants without selecting the wrong champion, leaking a late import, double-counting touch input, corrupting map state, or failing to restore focus and GPU work.

Local implementation update

The seven plans have now been implemented locally. A final council pass corrected two lifecycle gaps: replay now remounts the same canvas and browser WebGL context, and the modal isolates all current and dynamically added body siblings instead of relying on a fixed ID list. Sound was subsequently removed from production, the mock, tests, and plans; intentional silence is now an explicit invariant. A final visual pass replaced the simultaneous dissolve with a scene-ready card-and-connector cascade, raised the figures into rounded clay avatars, rebuilt the trophy with smooth curved geometry, and added a local 2026 WORLD CUP CHAMPIONS banner. A subsequent frontal rebuild aligned camera, tunnel, trophy, plinth, podium, team, flag, and banner to one centerline; changed the captain to a rear-to-front viewer-facing entrance; added articulated hands, named trophy grips, and deterministic two-bone arm targeting; lowered the overhead target to a physically reachable height; replaced lateral camera tracking with a centered push; consolidated the review mock onto the production scene; and kept the empty plinth dismissal during the carry.

What was discovered

The dashboard already has strong lifecycle patterns, but the celebration crosses render output, interactions, lazy imports, four trophy mounts, WebGL, byte-locked fixtures, accessibility, and responsive map state.

Why seven plans

Fewer plans would create a 3D mega-task no reviewer could meaningfully approve. More plans would split the same modules without independent acceptance. Seven matches the real dependency seams.

What not to do

Do not maintain a second mock renderer, clone the live WebGL canvas, add a framework, load remote models or media, parent the trophy to one hand, use an unreachable hero target, or treat a passing regenerated snapshot as proof.

Council influence

  • Karpathy: prove the smallest complete vertical slice first.
  • Russinovich: make renderer, timing, allocation, and failure costs explicit.
  • Willison/Hamel: test exact claims and distrust visual confidence.
  • Sean Grove: keep timing and interfaces in durable specs.
  • Hanselman/Litt: name boundaries so the owner can maintain them later.
  • Yegge/Naval: use small reviewable checkpoints and protect leverage.

Codebase Discovery

What the repository actually does today and which assumptions changed after inspection.

Static product contract

The site is GitHub Pages from docs/, with vanilla HTML, CSS, and ES modules. There is no frontend build or package runtime dependency. User data stays local; public results are generated by GitHub Actions.

Four bracket variants coexist

Mirrored Actual, mirrored My picks, Sideways Actual, and Sideways My picks are all rendered simultaneously. CSS chooses the visible pair via .brk-wrap[data-view][data-layout].

Champion identity is already correct

displayedChampion() keeps the user's champion before M104 and switches Actual to the real winner after M104. The trigger must read the active DOM box at activation time rather than duplicate scoring logic.

Lifecycle ownership is centralized

main.js destroys interactions, match details, and trophy instances before dashboard replacement. Celebration must join this lifecycle with its own generation token.

The trophy has mature safeguards

trophy.js already handles bounded passive rendering, intersection, visibility, reduced motion, phone fallback, resize, context loss, and disposal. The missing seam is explicit capture/suspension.

Render output is byte-locked

tests/map-frozen.mjs and tests/golden.mjs protect the mirrored markup. Adding one trigger attribute requires a deliberate fail-update-review workflow.

Important correction discovered

The baseline originally described pickBox() as shared by mirrored and Sideways output. Current source shows mirrored uses pickBox() and Sideways uses legacyPickBox(). The baseline and plans were corrected. This is a good example of why current source outranks stale planning prose.

Frontal rebuild decision record

Observation

The clay pass still read as a side-view scene. Camera, tunnel, trophy, plinth, team, flag, and banner used competing centerlines, and the hands only approximated trophy contact.

Evidence

Desktop and phone key frames exposed asymmetric staging and trophy drift. Rig measurements also showed the former 4.55 lift target exceeded the captain's upper-arm plus forearm reach. The standalone mock visibly differed from production.

Reasoning

Direct camera interaction requires one frontal axis. Physical contact requires placing the trophy first, reading real grip targets, and solving both arms afterward. Review accuracy requires one renderer, not two implementations that imitate each other.

Decision

Center every hero element at x = 0, move the captain rear-to-front, keep the trophy in scene space on a captain-owned carrier, add named left/right grips, solve both arms analytically, lower the lift to 3.68, and make the review harness call createChampionScene().

Tradeoffs

The frontal camera sacrifices some lateral parallax, and procedural avatars remain stylized rather than photoreal. In exchange, viewer interaction, symmetry, hand contact, phone framing, and production/review consistency become much stronger.

Lesson learned

A cinematic pose is not valid because it looks dramatic in isolation. It must be reachable by the rig, inspectable at key frames, and produced by the same code reviewers are approving.

Architecture Overview

The proposed production module boundaries and why they exist.

Why these modules

Each boundary corresponds to a distinct failure mode: trigger state, dashboard lifecycle, live trophy capture, renderer/resource ownership, model geometry, deterministic timing, bounded effects, and exact restoration. The scene owns the centered camera, trophy carrier, and arm solve; trophy geometry owns physical grip anchors; models own arm lengths and articulated hands. The center-out transition and generated championship banner remain inside those existing owners. Silence is a tested product contract, not another runtime module.

Why not more modules

The scene file is deliberately evolved across Plans 03-06. Splitting every builder into another file would increase context and disposal coupling without creating a separately verifiable handoff.

Learning Mode (ELI5)

The same architecture explained without assuming WebGL or browser-lifecycle experience.

Imagine a theater built inside the bracket page.

The champion box is a secret door. Four quick knocks open the theater. Before the curtain moves, the app takes a careful photo of where everything is: which bracket is showing, where the trophy is, whether the big table is expanded, where the page is scrolled, and which control has keyboard focus.

The app then puts a safe fence around the dashboard so people cannot click or tab into controls behind the show. It does not throw the dashboard away; it keeps the real bracket visible long enough for the left and right halves to slide apart like curtains. The cards nearest the center leave first, and each half of a connecting line travels with the card it touches, so the transition feels joined rather than like unrelated layers fading away.

The trophy is tricky. Copying the trophy canvas normally creates an empty picture, so the app must ask the trophy to draw one last frame and copy those pixels immediately. That picture settles onto the stage. Later, a separately built 3D version takes over while the captain's hand covers the switch.

Holding the trophy is like asking a person to grab two handles. First the app moves the trophy to the right place. Then it asks the trophy where its left and right handles are. Finally, it bends each arm toward one handle and curls the clay fingers. The trophy is not tied to one hand, and it never rises higher than the arms can physically reach.

One master clock tells the captain, camera, lights, flag, and confetti what moment they are in. If the browser tab is hidden, the clock stops. When the user returns, the show continues instead of jumping ahead.

When the show ends or is skipped, one cleanup routine removes the theater, stops graphics, and puts every dashboard detail back exactly where it was.

What a Senior Engineer Would Notice

Signals that distinguish a robust plan from a demo that merely works once.

Measurement order is part of correctness

Locking overflow can change page width. Source rectangles and trophy pixels must be captured before any stage insertion, class, inert, or overflow mutation.

Visibility is not lifecycle

Hiding a canvas does not stop RAF, WebGL, observers, or late promises. Every resource needs an explicit owner and destroy path.

One clock prevents drift

WAAPI, Three.js, and fallback timing cannot each use their own wall clock. They consume one logical visible-active time source.

Resource bounds are product decisions

DPR and instance counts are named constants because visual ambition without bounds becomes a reliability bug on phones.

Current source beats old documentation

The corrected pickBox ownership shows why an engineer verifies assumptions before encoding them in plans or abstractions.

Human review is still necessary

Tests can prove timing and cleanup, but only a person can approve acknowledgment readability, two-hand trophy contact, frontal centerline alignment, full-scale spatial cues, and composition.

One visual can have two owners

A connector joining independent cards cannot obey both transforms as one path. Split it at a stable midpoint so each half has one owner and one timing contract.

Physical reach is an acceptance criterion

An IK solver can clamp safely and still produce a visibly broken pose. Validate the desired prop target against measured arm lengths before solving.

A review surface should not imitate production

Duplicated cameras, rigs, and choreography inevitably drift. Keep review controls separate, but render the same production scene controller.

The feature's real leverage

The Easter egg itself is a one-off. The durable value comes from reusable seams: exact body-level modal restoration, a pure activation window, a paused logical clock, shared procedural trophy geometry, bounded scene infrastructure, and evidence-driven release practice.

Lessons Learned

Transferable engineering lessons from turning a creative brief into mechanical work.

Creative features still need boring foundations.

The more cinematic the outcome, the more valuable deterministic state, cleanup, and bounded resources become.

Plan around failure modes, not screen areas.

Trigger, capture, time, resources, models, effects, and evidence are better boundaries than "front end" versus "animation."

Review controls and rendering should have different owners.

The review may add play, scrub, team, and fallback controls, but camera, models, trophy contact, and choreography must come from production.

One source of truth prevents drift.

Phase times, stage heights, crowd limits, and fallback timing each need one named owner.

Browser and scene order-of-operations matter.

Measure before mutation, render before copying, pose the body before placing the trophy, place the trophy before reading grips, solve arms before rendering, pause before hiding, and restore before redrawing.

Tests and human review answer different questions.

Use pure tests for state/timing, source guards for browser-only construction, traces for cost, and human review for visual meaning.

Transition quality depends on ownership.

When cards, lines, trophies, and world layers meet, assign each moving visual one authoritative clock and transform owner.

Dramatic is not the same as mechanically valid.

Hero poses must fit the rig's real reach and preserve contact in every sampled frame, not only at the final still.

Final planning outcome
